# 前后端分离架构

## 目标

项目采用前后端分离，但“复杂度中心”明确放在后端：

- 后端：FastAPI + SQLite + 服务层 / 仓储层 / 存储抽象
- 前端：React + TypeScript + Vite

核心原则：

1. 后端负责认证、导入、加密、存储、播放与同步
2. 前端负责页面、交互、状态展示与用户流程
3. 双方通过稳定的 HTTP JSON API 协作

## 为什么这样拆

这个项目的高风险逻辑并不在界面，而在：

- Windows 主机路径导入
- `FFprobe` / `FFmpeg`
- 分片与 AES-256-GCM 加密
- 百度 OAuth 与远端对象访问
- HTTP Range 播放回放
- 本地缓存与远端回退

这些逻辑如果进入浏览器，会显著增加耦合、泄露主机能力边界，也会让安全模型变差。

## 后端负责什么

- Session 登录态
- 主机侧管理员设置
- 公开运行设置
- 视频导入与任务队列
- 封面抽取与本地加密封面管理
- 分片加密、manifest 写入与远端同步
- `mock` / `baidu` 存储适配
- 流式回放、seek 与 Range

## 前端负责什么

- 登录页
- 推荐页
- 媒体库页
- 视频详情页
- 播放页
- 管理页
- 设置页

前端不会处理：

- 内容密钥
- 分片解密
- 远端对象协议
- 主机文件系统访问

## `/admin` 与 SPA 的边界

当前有两套 UI 入口：

### `frontend/` SPA

面向正常使用流程，包含：

- `/recommend`
- `/library`
- `/manage`
- `/settings`
- `/videos/{id}`
- `/videos/{id}/play`

### 后端 `/admin`

面向主机管理员，使用 FastAPI 模板直接渲染：

- 管理员首次配置
- 百度开放平台凭据填写
- 会话密钥管理
- 播放回退下载并发
- 登录密码更新

这样拆分的原因是：

- `/admin` 需要在前端未准备好时也能独立可用
- 首次接入时，管理员先完成主机级配置，再进入前端配置页更自然

## 后端内部继续分层

- `route`：只处理 HTTP 参数、响应和鉴权
- `service`：处理业务编排
- `repository`：只做数据库访问
- `media`：只做媒体与加密相关纯能力
- `storage`：只做远端对象读写

这个结构的目的不是抽象层数本身，而是避免：

- 路由层直接拼百度请求
- 仓储层混入媒体逻辑
- 前端状态驱动后端内部实现
