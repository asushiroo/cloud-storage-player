# 本地导入与媒体探测

## 当前导入切片的目标

这一步并没有直接实现“上传百度网盘 + 分片加密”。

当前目标只有三个：

1. 验证本地主机路径导入链路可行
2. 记录导入任务状态
3. 生成最小视频元数据
4. 尽力抽取一张本地封面

也就是说，现在的导入能力本质上是：

> “把一个本地视频文件注册进目录数据库，并记录导入任务过程。”

## 为什么先做同步导入

最终架构当然更适合后台任务、可恢复导入、分阶段状态流转。

但当前阶段先做同步实现有几个好处：

1. 路径校验逻辑能先固定
2. `ffprobe` 集成能先打通
3. `import_jobs` 表和状态字段能先稳定
4. API 和前端后续不用重画入口

后面要升级成异步任务时，可以保留：

- 请求结构
- `import_jobs` 表
- 状态字段
- 视频元数据写入路径

## 导入服务当前流程

当前 `import_local_video()` 做的事情：

1. 校验 `source_path` 对应文件存在且是普通文件
2. 如果给了 `folder_id`，校验目录存在
3. 创建 `queued` 状态导入任务
4. 将任务推进到 `running`
5. 调用 `probe_video()` 执行 `ffprobe`
6. 根据探测结果写入 `videos` 表
7. 调用 `extract_cover()` 尝试抽取封面
8. 将任务标记为 `completed`
9. 如果 `ffprobe` 或其他步骤报错，则标记为 `failed`

## 媒体探测实现细节

当前使用外部命令：

```bash
ffprobe -v error -show_entries format=duration,format_name:stream=codec_type -of json <path>
```

当前解析结果只关心这些信息：

- 是否包含视频流
- 总时长 `duration`
- 格式名 `format_name`

另外两项元数据不依赖 `ffprobe`：

- `size`
  - 直接来自文件系统 `stat()`
- `mime_type`
  - 由文件扩展名通过 `mimetypes.guess_type()` 推断

## 当前导入不会做的事情

这一版导入**不会**：

- 切片
- 加密
- 上传百度网盘
- 生成 manifest
- 计算校验和

这些都留给后续导入阶段。

## 封面抽取实现细节

当前封面抽取使用外部命令：

```bash
ffmpeg -y -ss 0 -i <path> -frames:v 1 <output>.jpg
```

当前实现策略：

- 从视频开头抽取第一帧
- 输出为 JPG
- 文件保存在本地封面目录
- `videos.cover_path` 保存的是浏览器访问路径，例如 `/covers/12.jpg`

当前故障策略：

- 如果 `ffmpeg` 失败，导入任务仍然可以成功
- 只是该视频的 `cover_path` 会保持 `NULL`

## 当前失败语义

### 1. 请求级失败

如果 `source_path` 根本不存在，接口直接返回 `400`。

这种错误说明请求本身无效，不会创建成功任务。

### 2. 处理级失败

如果路径存在，但 `ffprobe` 失败或文件里没有视频流：

- 导入任务会进入 `failed`
- 接口仍返回任务详情

这样做是为了保留任务记录，方便后面追踪失败原因。

## 与未来目标架构的关系

当前导入切片已经把这些基础接口对齐了：

- 本地路径导入入口
- 服务层负责导入逻辑
- 数据库保留任务状态
- 视频元数据有固定落点

未来继续扩展时，优先建议按这个顺序推进：

1. 增加导入任务的阶段字段 / 更细状态
2. 增加分片描述
3. 增加 AES-GCM 加密
4. 增加 Baidu 上传与 manifest
5. 增加恢复与断点续传

## 关于 `/tmp` 测试视频

当前代码支持直接导入本机绝对路径，所以如果测试视频放在 `/tmp`，理论上可以直接调用：

```json
{"source_path":"/tmp/your-video.mp4"}
```

不过自动化测试并不依赖外部固定文件，而是运行时动态生成一个小样例视频。
这样测试更稳定，也不会依赖某个环境里恰好存在的文件。
